# 导入流程、Inpainting 提示与批量 Tag 输入设计

> 状态：讨论结论稿
>
> 日期：2026-07-19
>
> 适用平台：Windows / macOS
>
> 关联文档：[资源库管理、Tag 分类、设置与交互设计](./library-management-taxonomy-settings-design.md)

## 1. 目标

本文补充三个独立但能够共享导入基础设施的功能：

1. 识别 Inpainting 图片并提示其 metadata 无法精确复现原生成过程。
2. 支持拖拽图片以及 NovelAI 导出的 ZIP 压缩包进行导入。
3. 在当前 Prompt 作用域内一次添加多个 Tag。

这些功能不改变“结果图不可变”的原则。导入图片后，软件仍然保存图片已经发生的生成事实；提示只说明这些事实是否足以复现生成过程。

## 2. 本地样本检查结果

### 2.1 Inpainting 图片

本地验证样本：

`E:\Works\AI\AIPictureNovelAI\明日方舟\明日方舟第七十期\0.png`

该 PNG 可以被现有 metadata 解析器正常读取，包含 Prompt、Undesired Content、Seed、Sampler、Steps、CFG 和模型信息。用于识别局部重绘的关键字段为：

```text
request_type: "NativeInfillingRequest"
```

同时该样本中的 `img2img` 为 `null`，因此不能仅用 `img2img` 判断是否为 Inpainting。现有软件已经把 `request_type` 保存在原始扩展 metadata 中，但没有把它规范化为生成模式，也没有在界面上提示。

### 2.2 NovelAI 导出 ZIP

本地验证样本：

`E:\Works\AI\AIPictureNovelAI\NAI-images_(2026-01-12_21-52-29)_1-of-1.zip`

只读检查结果：

- 共 80 个条目，全部为 PNG。
- 解压后总大小约 142 MB。
- 没有子目录。
- 没有绝对路径或包含 `..` 的危险路径条目。
- PNG 已经压缩，因此 ZIP 中压缩前后体积基本相同。

该结构适合直接进入现有图片 metadata 导入流程。自动化测试不能依赖上述用户个人路径，也不能把原始大文件提交到仓库；实现时应从样本提取最小化、无私人内容的测试 fixture。

## 3. Inpainting 识别与提示

### 3.1 数据语义

为图片生成信息增加规范化的生成模式：

- `text_to_image`
- `image_to_image`
- `inpainting`
- `unknown`

第一版至少将 `request_type === "NativeInfillingRequest"` 映射为 `inpainting`。原始字段继续完整保留，规范化字段只用于查询、提示和后续分支判断。

识别规则必须以明确 metadata 为主，不能因为图片尺寸、透明度、文件名或 Prompt 内容“看起来像局部重绘”就作出推测。遇到未来未识别的 `request_type` 时保留原值并标记为未知，便于后续扩展。

### 3.2 用户提示

Inpainting 不是导入错误，图片仍然可以正常进入作品库。推荐提示文案：

> 此图片由局部重绘生成。Metadata 包含最终 Prompt 和部分生成参数，但不包含完整的原始底图与蒙版，因此无法精确复现原生成过程。你仍然可以查看、复制 Prompt，或基于现有参数创建近似分支。

提示出现于：

1. 导入完成摘要：显示“局部重绘 · 无法精确复现”状态。
2. 图片信息区：使用固定状态徽标和可展开说明。
3. 从结果创建分支时：再次说明新分支只能继承已知参数，不能声称精确复现。
4. 对比实验中：允许参与视觉对比，但不能被标记为严格可复现实验基准。

提示使用警告状态而不是错误状态，不阻止导入、复制 Prompt、分类、翻译、收藏或加入系列。

### 3.3 可复现性状态

“生成模式”和“可复现性”是两个维度。后续可以统一显示：

- 参数完整，可按 metadata 重建
- 局部重绘，缺少底图或蒙版
- metadata 不完整
- metadata 缺失
- metadata 存在但格式未知

这样能避免只为 Inpainting 写一个孤立判断，并能直接服务于实验和分支模型。

### 3.4 验收标准

- 样本 `0.png` 被识别为 `inpainting`。
- Prompt、Seed 和其他已有 metadata 仍然正常显示。
- 界面明确说明“不能精确复现”，但不把图片标记为损坏。
- 创建分支时保留来源和不完整原因。
- 普通 text-to-image 图片不出现误报。
- 未知 `request_type` 不导致导入失败。

## 4. 拖拽导入图片

### 4.1 拖拽入口

在作品库页面中，用户可以把文件拖到：

- 整个作品库工作区：导入到当前库。
- 某个收藏集：导入后同时加入该收藏集。
- 明确的空状态区域：作为首次导入入口。

拖拽进入窗口时显示全局导入覆盖层，明确列出可接受类型：PNG 和 ZIP。无效文件使用拒绝状态，不能让 Electron 导航到本地文件。

点击“导入”按钮和拖拽必须调用同一套导入命令，不能维护两条行为不同的实现。

### 4.2 多文件导入

允许一次拖入多张图片。导入流程分为：

```text
接收文件
    → 类型与大小预检
    → 计算文件指纹并检查重复
    → 复制到应用管理目录
    → 读取 metadata 并生成缩略图
    → 单项写入数据库
    → 显示批次摘要
```

导入过程中显示已完成数量、当前文件和取消操作。一个文件失败不应阻止其他有效文件；失败项保留具体原因，批次结束后统一报告。

重复判断以内容指纹为主，不依赖文件名。遇到相同内容时默认跳过重复导入，并在摘要中提供“查看已有作品”。以后如果需要允许同文件多份注释，应通过单独的产品决策实现，不能绕过指纹检查制造重复资产。

### 4.3 导入批次

每次拖入或 ZIP 导入创建一个临时批次记录，便于：

- 查看本次成功、跳过和失败数量。
- 一键筛选本次导入的作品。
- 撤销本次导入；已有引用的作品需要单独确认。
- 将本批次保存为收藏集。

默认不自动创建永久收藏集，避免长期导入后侧栏堆积大量以 ZIP 文件名命名的分组。

## 5. 拖拽 ZIP 一键导入

### 5.1 支持范围

第一版只支持标准 `.zip`，只导入其中通过文件签名验证的 PNG。忽略 macOS 资源目录、隐藏系统文件和不支持的其他条目，并在摘要中说明。

不支持：

- ZIP 内嵌套 ZIP。
- 加密或需要密码的压缩包。
- 7z、RAR 等其他格式。
- 伪装成 PNG 的其他文件。

未来扩展格式时继续复用同一批次导入协议。

### 5.2 安全预检

在读取图片前检查：

- 拒绝绝对路径、盘符路径和包含 `..` 的路径穿越条目。
- 不创建或跟随压缩包中的符号链接。
- 限制条目数量、单文件展开大小、总展开大小和异常压缩比。
- 不把整个压缩包一次性解压到内存。
- 不直接解压到项目目录或最终资源目录。
- 逐条流式读取，验证 PNG 签名后再进入图片导入流程。

具体数值上限在实现阶段根据大样本测试确定。超过安全上限时给出明确原因和建议，不能只显示“导入失败”。

### 5.3 一键体验

对于类似验证样本的标准 NovelAI ZIP，拖入后无需逐张确认，直接开始导入。界面流程：

```text
拖入 ZIP
    → 快速预检并显示“发现 80 张 PNG”
    → 自动开始导入
    → 显示进度、重复和异常数量
    → 完成后打开本次导入筛选
```

特别大的压缩包在预检后需要显示预计磁盘占用和确认；普通压缩包保持一键体验。

### 5.4 中断与恢复

每张图片完成后独立提交，取消操作只停止尚未开始的条目。已完成项保留在本次导入批次中，并提供“撤销已导入项目”。应用异常退出后，可以识别未完成批次并允许继续或清理临时数据。

### 5.5 验收标准

- 验证样本能够识别出 80 张 PNG。
- 导入结果与逐张导入使用完全相同的 metadata 解析规则。
- 相同 PNG 不会因为 ZIP 内文件名不同而重复导入。
- 非法路径、嵌套压缩包和伪 PNG 不会写入资源目录。
- 用户可以取消，并看到成功、跳过、失败和未处理数量。
- 导入大量图片时界面保持可操作，不一次加载全部原尺寸图片。

## 6. 批量添加 Tag

### 6.1 输入行为

当前单 Tag 输入框升级为批量输入框，支持以下分隔符：

- 英文逗号 `,`
- 中文全角逗号 `，`
- 换行

首尾分隔符、连续分隔符和只包含空白的片段会被忽略。例如：

```text
,akakura,ciloranko，white background
```

解析结果为：

1. `akakura`
2. `ciloranko`
3. `white background`

这也修复了部分已有 Prompt 使用中文逗号时无法正确拆分的问题。

### 6.2 Prompt 语法

批量解析继续复用统一 Prompt 解析器，不能单独使用简单的字符串 `split`。解析器需要：

- 正确处理 `1.3::shirt dress ::` 等权重语法。
- 保留原始顺序。
- 保留无法规范化的原始片段和语法诊断。
- 不把孤立的 `::` 当作正常 Tag。
- 只把 Tag 加入当前选中的 Base Prompt、Base Undesired Content 或 Character 作用域。

完全相同的 Tag 不静默删除。重复项可以添加，但在确认信息中显示重复数量，避免软件擅自改变用户希望表达的 Prompt 强度。后续可提供“跳过重复”或“合并为显式权重”的整理命令。

### 6.3 交互

- 输入中包含多个 Tag 时，按钮显示“添加 3 个”。
- Enter 提交；Shift+Enter 插入换行。
- 提交后显示添加数量、语法警告数量和重复数量。
- 添加操作作为一个整体进入撤销记录，一次撤销即可移除这一批 Tag。
- 大批量输入时先显示解析预览，避免误把整段自然语言拆成大量 Tag。

### 6.4 验收标准

- 示例输入一次添加 3 个 Tag。
- 中文与英文逗号可以混用。
- 空片段不会生成空 Tag。
- 权重语法保持正确。
- 添加到 Character Prompt 时不会误入 Base Prompt。
- 一次撤销能够恢复添加前状态。

## 7. 优先级与实施顺序

### 7.1 三项功能内部优先级

| 顺序 | 功能 | 优先级 | 原因 |
| --- | --- | --- | --- |
| 1 | Inpainting 识别与不可精确复现提示 | P0 | 涉及生成信息真实性，且样本已有明确字段，范围小、价值高 |
| 2 | 批量添加 Tag 与全角逗号解析 | P1 快速改进 | 独立、改动范围小，能立即改善高频编辑操作 |
| 3 | 拖拽导入单张/多张图片 | P1 | 是作品库管理和 ZIP 导入共用的入口与基础协议 |
| 4 | 拖拽 ZIP 批量导入 | P1，依赖第 3 项 | 用户价值高，但需要安全预检、进度、取消、重复检测和批次恢复 |

ZIP 导入不应直接写成一个独立的“解压后循环导入”功能。先建立统一图片导入命令和批次模型，ZIP 只负责安全地提供图片流。

### 7.2 放入总体路线图

推荐把此前总体路线图调整为：

1. **数据正确性快速改进**：Inpainting 提示。
2. **编辑快速改进**：批量添加 Tag、全角逗号解析。
3. **作品资产管理基础**：收藏集、多选、回收站、引用检查、统一导入批次。
4. **导入体验**：拖拽图片，然后接入 ZIP。
5. **不可变结果与可编辑注释**。
6. **Vibe 管理与 Information Extracted 来源状态**。
7. **设置、Tag 分类和 AI 缓存版本**。
8. **分支、系列与对比实验**。
9. **高级智能、对比和整体动效完善**。

前两项可以作为开始资产管理重构前的短周期交付，不改变核心数据库方向。主要工程仍应从作品资产管理基础开始；ZIP 导入应在导入批次和安全删除规则建立后完成。

## 8. 跨平台验证

Windows 与 macOS 都需要验证：

- 从资源管理器 / Finder 拖入单张、多张图片和 ZIP。
- 拖入无效文件时应用不会导航离开或冻结。
- 含中文、日文、空格、括号和长文件名的 ZIP 条目。
- 导入取消、应用退出和再次启动后的批次恢复。
- Ctrl / Cmd、Enter / Return 和 Shift+Enter 行为。
- 高对比度、减少动态效果和键盘完整操作。

本地样本路径只用于 Windows 手动验证。如果实现后需要 macOS 实机确认，再通过 `coordination/MACOS.md` 创建包含提交、动作和预期结果的交接项。
