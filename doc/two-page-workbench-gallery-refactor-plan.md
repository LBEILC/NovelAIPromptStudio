# 工作台与图片库两阶段重构计划

> 状态：两个阶段均已实施
> 日期：2026-07-22  
> 适用平台：Windows / macOS，尽量保持 Linux 兼容  
> 视觉规范：[Lobe 视觉层级与颜色语义规范](./lobe-visual-language.md)

## 1. 文档目标

NovelAI Prompt Studio 回到最初目标：方便地从 NovelAI 图片中读取并编辑 Tag，然后复制到 NovelAI 中继续生成和调试。

重构后的应用只有两个主要页面：

1. **工作台**：图片拖入后立即解析 Tag，并直接进入编辑模式。
2. **图片库**：图片拖入后保存到本地，供以后浏览和再次送入工作台。

设置入口仍可保留在侧边栏底部，但它属于应用级辅助入口，不算第三个业务页面。

本计划取代以下旧产品方向；旧文档只保留为历史记录：

- [图库、Vibe 库与 Tag 库三阶段开发计划](./three-library-information-architecture-development-plan.md)
- [资源库管理、Tag 分类、设置与交互设计](./library-management-taxonomy-settings-design.md)
- [图片系列、对比实验与生成分支设计](./experiment-and-branch-design.md)

## 2. 产品定位

产品不再尝试成为完整的 NovelAI 创作资产管理器或本地生成实验系统，而是：

> 一个极简的 NovelAI 图片收藏库，以及一个由图片驱动的 Tag 编辑器。

功能是否保留只看两个条件：

1. 是否直接服务于“读取、编辑、复制 Tag”或“保存图片”。
2. 用户是否能在本应用内确认操作结果。

必须回到 NovelAI 才能看到效果的调参功能不在本应用中重复实现。应用不负责生成图片，也不尝试模拟 NovelAI 的生成反馈。

## 3. 已确认的产品边界

### 3.1 保留

- 从 NovelAI PNG 中解析 Prompt Tag。
- 在工作台中添加、删除、排序和修改 Tag。
- 编辑 Tag 权重和支持的 Prompt 结构。
- 在工作台中使用 AI 翻译 Tag，并查看或调整 Tag 分类。
- 复用已保存的 Tag 翻译与分类结果，避免重复请求 AI。
- 复制编辑后的 Prompt，供用户粘贴到 NovelAI。
- 从图片 metadata 中解析 Vibe。
- 只读显示、复制或导出图片中能够恢复的 Vibe 数据。
- 将图片导入本地图片库。
- 浏览、打开、从图片库移除图片。
- 从图片库把图片送入工作台。
- 必要的主题、窗口、平台和 OpenAI-compatible API 设置。

### 3.2 删除

- Tag 库页面，以及 Tag 详情、备注、显示状态和图片反向引用等资源管理能力。翻译与分类仍保留在工作台 Tag 编辑器中。
- Vibe 库、`.naiv4vibe` 资源管理、来源管理和反向引用。
- Vibe 强度、Information Extracted 等本地编辑能力。
- Prompt 版本历史和版本恢复。
- 生成分支、分支草稿、待生成状态和结果匹配。
- 创作系列、对比实验、关系分组和关系对比。
- 收藏集、自动分组和其他非必要图库分类能力。
- 为上述功能服务的批量操作、上下文菜单、数据库表、IPC、测试和样式。
- 与工作台 Tag 翻译、分类无关的其他 AI 功能。

删除页面或入口不等于立即物理删除用户旧数据。数据处理规则见第 8 节。

### 3.3 明确不做

- 不在本地预览 Vibe 强度变化的生成效果。
- 不修改图片文件中的原始 metadata。
- 不把工作台编辑结果写回图片库中的原图记录。
- 不建立 Tag、Vibe、图片或 Prompt 之间的关系图谱。
- 不提供分支、版本树、实验、参数差异或结果回传流程。
- 不在本轮重构中加入多工作台、多标签页或云同步。

## 4. 目标信息架构

```text
SideNav
├─ 工作台（默认页）
├─ 图片库
└─ 设置（底部辅助入口）
```

两个页面必须使用不同且清楚的拖放语义：

| 拖放位置 | 用户意图 | 应用行为 | 是否进入图片库 |
| --- | --- | --- | --- |
| 工作台 | 现在编辑这张图片的 Tag | 读取图片、解析 metadata、打开编辑器 | 否 |
| 图片库 | 保存这张图片 | 复制到本地资产目录、生成缩略图、建立图片记录 | 是 |

拖拽覆盖层必须直接说明结果：

- 工作台：`松开以解析并编辑，不会保存到图片库`
- 图片库：`松开以导入并保存到图片库`

同一个动作不能因为当前页面不明确而产生不同结果。拖拽进入窗口后，应通过页面范围、覆盖层文案和图标共同说明当前目标。

## 5. 页面设计

### 5.1 工作台

工作台是应用默认页，也是核心页面。

#### 空状态

- 页面中央提供明确的图片拖放区域。
- 主文案：`拖入 NovelAI 图片开始编辑 Tag`。
- 辅助文案：`图片只用于本次工作台解析，不会加入图片库`。
- 同时提供标准文件选择按钮，保证键盘用户和不方便拖拽的用户可以完成相同操作。

#### 编辑状态

页面使用稳定的双栏结构：

- 左侧：当前图片预览和最少量的来源信息。
- 右侧：Tag 编辑器，载入后立即可编辑，不再经过“概览”或“进入编辑”步骤。

Tag 编辑器保留：

- Base Prompt 与 Character Prompt 等图片中实际存在的 Prompt 区域。
- Prompt 与 Undesired Content 的必要区分。
- Tag 新增、删除、排序和权重编辑。
- 对当前图片中的 Tag 执行 AI 翻译。
- 查看和调整 Tag 分类；分类用于帮助阅读与编辑，不改变 Prompt 原文。
- 复用按规范化 Tag 原文保存的翻译与分类结果。
- 原始语法的可靠解析与重新序列化。
- `复制 Prompt` 主操作。
- `恢复为图片原始内容` 次要操作，仅作用于当前工作台会话，不形成历史版本。

翻译与分类属于编辑辅助信息，而不是新的资源库：

- 不建立独立 Tag 库页面或 Tag 详情页。
- 翻译和分类可以跨工作台会话复用，但不保存图片关系、使用次数或反向引用。
- AI 操作只处理当前工作台中的 Tag，不提供全库批量整理流程。
- 复制 Prompt 时始终使用 Tag 原文；翻译和分类不能改写复制结果。
- OpenAI-compatible API 的地址、模型、凭据和连通性测试继续由设置入口管理，并保持现有安全存储边界。

Vibe 只作为只读附属信息：

- 没有 Vibe 时不占用固定大区域。
- 检测到 Vibe 时显示数量和可恢复状态。
- 只提供复制或导出原始数据的操作。
- 不显示可编辑的强度、来源、版本或关系字段。
- PNG metadata 不能恢复来源参考图时，必须明确说明，不能暗示数据完整。

#### 会话边界

- 工作台一次只编辑一张图片。
- 拖入新图片会替换当前工作台内容。
- 当前编辑状态可以在本机自动恢复，但它只是单一工作台草稿，不进入图片库，也不形成版本历史。
- 替换、清空或恢复原始内容时，应避免无提示丢失尚未复制的修改。

### 5.2 图片库

图片库只负责保存和重新找到图片，不承担 Prompt、Vibe 或实验管理职责。

#### 空状态

- 页面提供图片拖放区域和文件选择按钮。
- 主文案：`拖入图片以保存到图片库`。
- 辅助文案明确图片会被复制到本地应用数据目录。

#### 图片网格

- 以图片为视觉主体，使用简单、自适应的缩略图网格。
- 卡片只显示识别图片所需的信息，不展示关系数、分支数、实验状态或大量参数徽章。
- 支持打开预览、在文件管理器中显示、送入工作台和从图片库移除。
- 右键菜单的主要业务操作为 `在工作台中打开`。
- 可以支持将图库图片直接拖到工作台入口，但它必须与右键操作产生相同结果。

#### 图片预览

- 预览以图片本身为主。
- 可只读查看原始 Prompt 和已解析 Vibe 摘要。
- 编辑操作统一跳转到工作台，不在图片详情中建立第二套编辑器。
- 不显示关系、方案、实验、系列或参数对比入口。

## 6. Lobe UI 设计规范

本轮重构继续使用现有 Lobe UI 技术栈和视觉规范，不另建组件系统。

### 6.1 继续沿用

- `LobeProvider`、现有主题切换和 Lobe/Ant Design token。
- Slate 中性色与克制的蓝色主色。
- 本地打包的 Geist、Geist Mono 和 HarmonyOS Sans SC 字体。
- [Lobe 视觉层级与颜色语义规范](./lobe-visual-language.md)定义的表面、按钮、状态色和选择态规则。
- Lobe 的 Button、ActionIcon、Alert、Empty、Input、Segmented、Tooltip、Popover 和 Context Menu 等基础组件。

### 6.2 页面应用原则

- 工作台是连续工作表面，不使用卡片嵌套卡片。
- 图片是图片库的视觉主体，文件名和操作降为次级信息。
- 每个页面最多只有一个高强调主操作。
- 蓝色只表达主操作、当前选择和焦点，不装饰普通文字。
- 拖放状态使用完整覆盖面或完整边框，不使用单侧色条。
- 动效只用于拖放反馈、页面状态变化和操作确认，不加入装饰性动画。
- 业务 CSS 只负责必要布局与尺寸，不重复实现 Lobe 组件已有的交互状态。

旧视觉规范中涉及 Vibe 库、Tag 库、分支或实验的业务示例不再适用；其基础视觉原则继续有效。

## 7. 两阶段实施计划

### 第一阶段：建立工作台并锁定核心编辑闭环（已实施）

#### 目标

用户可以把一张 NovelAI 图片拖入应用，立即看到并编辑其中的 Tag，随后复制 Prompt；图片不会因此进入图片库。

#### 实施内容

1. 将一级导航收敛为工作台和图片库，并让工作台成为默认页。
2. 建立独立工作台会话模型，不再直接修改图库图片记录。
3. 接入 PNG metadata 读取与错误反馈。
4. 复用并拆分现有 Prompt/Tag 解析和编辑能力，移除资源库、版本与分支耦合。
5. 将 AI 翻译、分类和缓存能力收敛到工作台 Tag 编辑器，并保留对应 API 设置。
6. 实现工作台空状态、拖放覆盖层、文件选择和图片预览。
7. 图片载入完成后直接聚焦 Tag 编辑区。
8. 实现 Prompt 复制和恢复图片原始内容。
9. 以只读方式显示并复制/导出解析到的 Vibe。
10. 为工作台草稿提供最小的本机恢复能力，但不建立历史记录。
11. 隐藏 Tag 库、Vibe 库、分支、实验、关系和对比入口。

#### 验收标准

- 从 Explorer 和 Finder 拖入受支持 PNG 后能够进入编辑状态。
- 工作台拖入不会在图片库中新增记录或复制图片文件。
- Tag 的增加、删除、排序、权重和 Prompt 复制结果正确。
- 当前工作台 Tag 可以翻译和分类，缓存结果重启后仍可复用，且不改变 Prompt 原文。
- 解析失败时保留当前工作台内容，并显示可执行的错误信息。
- 含 Vibe 的图片只显示只读复制/导出操作，不存在强度编辑控件。
- 键盘用户可以通过文件选择完成与拖拽相同的流程。
- Windows 和 macOS 的窗口、拖拽、快捷键和文本布局正常。

### 第二阶段：简化图片库并删除旧功能链（已实施）

#### 目标

用户可以把图片拖入图片库持久保存，并能随时将已保存图片送入工作台；旧的资源库和实验体系从产品及代码中移除。

#### 实施内容

1. 将图库改为只承担导入、浏览、预览、移除和送入工作台。
2. 明确图库拖放覆盖层和持久化结果。
3. 让 `在工作台中打开` 成为图片主要上下文操作。
4. 删除 Tag 库和 Vibe 库页面、路由及状态。
5. 删除 Vibe 编辑，以及 Tag 备注、显示状态和图片反向引用；保留工作台所需的 Tag 翻译、分类与缓存。
6. 删除分支、分支结果匹配、系列、实验、关系和对比功能。
7. 删除对应 IPC、数据库读写、前端工具模块、测试、菜单和样式。
8. 清理不再使用的依赖和资源。
9. 按第 8 节迁移现有用户数据。
10. 更新 README、测试说明和历史设计文档状态。

#### 实施结果

- 图片库已收敛为导入、搜索、排序、预览、文件定位、移除和送入工作台。
- Tag/Vibe 库、分支、版本、系列、实验、关系、对比及其前端状态、IPC、数据库运行时接口、工具模块、测试和样式已移除。
- Vibe 仅保留图片 metadata 解析、只读展示和编码复制。
- Tag 字典仅保留翻译、分类和来源字段，不再提供独立浏览、备注、可见性或反向引用。
- 现有 `studio.sqlite` 首次由第二阶段版本打开前，会在同目录创建一次不可覆盖的 `studio.pre-phase2.sqlite` 备份。
- 旧表不会在升级时被删除；运行时不再查询或写入这些表。旧 Vibe 资产也不会被自动清理。

#### 验收标准

- 图库拖入后图片被复制到应用资产目录，并在重启后继续存在。
- 同一图片重复导入时行为明确，不产生无法解释的重复记录。
- 图库图片可以通过右键或等价可访问操作送入工作台。
- 从图库打开工作台不会修改图库中的原始 Prompt 或 metadata。
- 应用中不存在 Tag 库、Vibe 库、强度编辑、分支、实验、关系或对比入口。
- 工作台中的 AI 翻译、Tag 分类和对应 API 设置仍可正常使用。
- 删除旧模块后所有测试和生产构建通过。
- Windows 完成启动、拖放、文件对话框和文件管理器操作验证；macOS 按需记录并完成 Finder 与原生窗口交接验证。

## 8. 数据迁移与删除策略

重构目标是删除产品复杂度，不是静默销毁用户数据。

1. 升级前对现有数据库创建带版本或时间戳的备份。
2. 已导入的图片文件、缩略图、原始 Prompt 和原始 metadata 必须保留。
3. 工作台始终从图片原始数据建立独立草稿，禁止覆盖图片事实。
4. 旧 Tag/Vibe 注释、分支、系列和实验记录在首个重构版本中停止使用，但不要求迁移时立即物理擦除。
5. 旧 Vibe 资产文件不能在没有明确迁移或导出策略时自动删除。
6. 当新版本稳定且备份路径明确后，才可以在后续维护中清理废弃表和孤立资产。
7. 数据库降级或迁移失败时，应用必须停止写入并给出备份位置，不能以空库继续运行。

## 9. 技术收敛方向

目标数据模型只需要两个业务概念和一个辅助索引：

- `WorkbenchSession`：当前图片引用、原始解析结果、当前编辑结果和更新时间。
- `LibraryImage`：本地图片路径、缩略图路径、导入时间和只读原始 metadata。
- `TagAnnotation`：以规范化 Tag 原文为键，只保存可复用的翻译、分类及必要的 AI 缓存信息。

Tag 是 `WorkbenchSession` 中的结构化内容，不再是可浏览的独立资源。`TagAnnotation` 只为编辑器提供翻译和分类，不形成 Tag 库。Vibe 是图片解析结果中的只读数据，不再是可编辑资源。

建议按以下边界拆分代码：

- `WorkbenchPage`：页面状态、拖放和编辑闭环。
- `TagEditor`：纯 Prompt/Tag 编辑组件，不知道图库、分支或数据库。
- `TagAnnotationService`：负责翻译、分类、缓存和安全的 AI 主进程调用，不提供资源库页面。
- `VibeReadOnly`：只读摘要与复制/导出。
- `GalleryPage`：持久导入和图片网格。
- `ImagePreview`：图片预览和送入工作台入口。
- `metadata` / `prompt`：保留无 UI 依赖的解析与序列化函数。

不再让单个页面组件同时管理图库、资源索引、分支、实验、关系、比较和编辑状态。

## 10. 完成定义

两阶段全部完成后，应满足：

- 主导航只有工作台与图片库两个业务页面。
- 用户从启动应用到编辑 Tag 不需要经过详情页或模式切换。
- 工作台拖入不保存，图片库拖入才保存，且界面始终明确说明结果。
- Vibe 只能从图片解析并只读复制/导出，不能编辑强度。
- AI 翻译和 Tag 分类保留在工作台中，且不会改写 Prompt 原文。
- Tag/Vibe 资源库、版本、分支、系列、实验、关系和对比能力全部退出产品。
- 原有 Lobe UI 视觉语言、跨平台约束和可访问性交互得到保留。
- `npm ci`、`npm test` 和 `npm run build` 通过。
