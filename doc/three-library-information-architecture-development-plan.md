# 图库、Vibe 库与 Tag 库三阶段开发计划

> 状态：已确认方案，待实施  
> 日期：2026-07-19  
> 适用平台：Windows / macOS  
> 关联文档：[资源库管理、Tag 分类、设置与交互设计](./library-management-taxonomy-settings-design.md)  
> 关联文档：[图片系列、对比实验与生成分支设计](./experiment-and-branch-design.md)  
> 视觉规范：[Lobe 视觉层级与颜色语义规范](./lobe-visual-language.md)

## 1. 文档目标

当前界面同时存在左侧作品列表、中央图片/Prompt 工作区、右侧 Prompt/Vibe/参数检查器和底部分支/实验胶片。每个区域都承担导航和编辑职责，用户需要同时理解多个选择状态，且右侧固定宽度和底部胶片会持续压缩图片与 Prompt 的可用空间。

本计划将应用重构为三个资源导向的大页面：

1. **图库**：导入、浏览、筛选和管理图片；点击图片进入图片详情。
2. **Vibe 库**：管理来源图片、`.naiv4vibe` 编码和参数变体，并反向查看使用图片。
3. **Tag 库**：管理 Tag 的翻译、分类、显示状态和 AI 缓存，并反向查看使用图片。

永久右侧检查器和底部胶片从主交互中删除。它们承载的 Prompt、Vibe、元数据、分支、实验和对比能力迁移到独立详情页和关系分组中。

本文分三个可独立验收的阶段实施，避免一次性替换全部界面造成数据能力或工作流回退。

## 2. 三阶段总览

| 阶段 | 主要目标 | 用户可见结果 |
| --- | --- | --- |
| 第一阶段 | 建立三页面外壳、图库网格和图片详情 | 使用 Lobe `SideNav` 切换资源库；图库变为图片网格；图片详情通过页内 Tabs 查看 Prompt、Vibe 和元数据；永久右侧检查器消失 |
| 第二阶段 | 建立 Vibe 库、Tag 库和双向索引 | 可以从 Vibe/Tag 反向查看图片；Tag 原文只读，仅翻译、分类和显示状态可编辑；问题 Tag 默认隐藏但不改变 Prompt |
| 第三阶段 | 建立图片关系分组并完成旧交互退役 | 关系页按“同 Prompt、不同 Seed”等规则分组展示图片；分支与实验迁入普通网格；底部胶片及旧布局代码完全删除 |

## 3. 已确认的产品原则

### 3.1 资源与事实

1. 图片及其原始 Prompt、Vibe、Seed、模型和生成参数是不可变生成事实。
2. 修改生成配置必须创建新的生成方案或分支，不能覆盖原图事实。
3. 翻译、分类、备注、收藏和显示偏好属于用户注释，可以修改。
4. 资源库负责“查找和管理资源”，关系页负责“解释图片之间为什么相关”。

### 3.2 Tag

1. Tag 原文本体永久只读，Tag 库不提供重命名、替换或删除原文的能力。
2. 允许修改中文翻译、主分类、备注、搜索别名、显示状态和 AI 校正状态。
3. 问题 Tag 可以默认隐藏，但隐藏只影响界面，不改变图片 Prompt 和原始 metadata。
4. 图片详情中的问题 Tag 放入默认折叠的“异常 Tag”分组，不能完全消失。
5. 复制原始 Prompt 时仍包含隐藏 Tag；只有用户明确按当前筛选复制时才允许排除，并显示实际复制数量。

### 3.3 Vibe

1. `.naiv4vibe` 文件及其编码保持只读，不因界面操作被重新编码或改写。
2. 来源 PNG、编码文件、Information Extracted 位置和使用参数分别表达，不混成同一个状态。
3. 用户手动声明的 Information Extracted 标记为“用户设置、未验证”。
4. Vibe 库必须能够反向列出使用该来源或编码的图片。

### 3.4 旧交互

1. 永久右侧检查器删除，其内容迁入图片详情页。
2. 底部胶片及横向滚动交互删除，不再继续开发自定义胶片组件。
3. 分支、实验、系列和对比的数据能力保留，但使用标准列表、网格和独立页面呈现。
4. 旧胶片在新入口具备功能对等后才删除代码和样式，避免历史分支或实验失去入口。

│  └─ 图片详情

│     ├─ Vibe
│     ├─ 元数据
│     └─ 关系与方案
├─ Vibe 库
│  ├─ 来源与编码列表
│  └─ Vibe 详情
│     ├─ 来源 PNG
│     ├─ 编码与 Information 位置
│     ├─ 使用配置
│     └─ 相关图片
├─ Tag 库
│  ├─ Tag 词典列表
│  └─ Tag 详情
│     ├─ 翻译与分类
│     ├─ 显示与异常状态
│     ├─ 使用范围

### 4.1 一级导航

- 桌面端使用 Lobe `SideNav`，不使用移动端 `TabBar`。
- 一级导航只包含图库、Vibe 库和 Tag 库。
- 设置放在 `SideNav.bottomActions`。
- 图标必须带 Tooltip 和可访问名称；不能只依赖图形表达页面含义。
- 当前页面使用完整选中表面或完整边框，不使用单侧主题色条。

### 4.2 页面历史

- 从图库进入图片详情后，返回必须恢复原筛选、滚动位置和多选状态。
- 从 Vibe/Tag 详情点击相关图片后，返回必须回到原 Vibe/Tag 详情。
- 页面状态使用明确的路由或等价导航状态，不继续依赖多个互相耦合的布尔值。
- `Esc`、平台返回快捷键和面包屑行为保持一致。

## 5. 数据边界与索引

### 5.1 Tag 原始出现记录

每张图片中的 Tag 出现记录至少保留：

- 原始文本与原始语法。
- Prompt 区域：Base 或 Character。
- 极性：Prompt 或 Undesired Content。
- Character 序号和 Position。
- 权重、顺序和语法警告。
- 来源图片和导入 metadata。

这些字段只读。Tag 库中的编辑不能回写这些记录。

### 5.2 Tag 词典记录

Tag 词典以稳定查找键关联多个原始出现记录，维护：

- 中文翻译及来源：AI、人工或缓存。
- 主分类及来源：AI、人工或缓存。
- 备注与仅用于搜索的别名。
- `visible / hidden / problematic` 显示状态。
- 问题原因，例如空内容、孤立 `::`、不完整强调语法或解析异常。
- 首次发现、最后使用、使用次数和关联图片数量。

词典记录被归档或隐藏时，原始出现记录不删除。

### 5.3 Vibe 关联索引

Vibe 需要区分两个指纹：

1. **来源指纹**：来源 PNG 内容哈希或编码内容哈希，用于确认是否为同一个 Vibe 来源。
2. **使用指纹**：来源指纹 + Information Extracted + Reference Strength 等使用参数，用于确认是否为同一套使用配置。

图片、来源 PNG、`.naiv4vibe` 变体和项目内使用记录建立可查询关联。缺少来源 PNG 时仍保留编码身份，并显示明确警告。

### 5.4 图片关系指纹

第三阶段使用以下稳定指纹生成关系：

- **Base 指纹**：Base Prompt + Base Undesired Content，保留顺序、权重和语法。
- **Character 指纹**：每个 Character 的 Prompt + Undesired Content + Position。
- **完整 Prompt 指纹**：Base、全部 Character、Character 顺序、权重和位置。
- **Vibe 来源指纹**与**Vibe 使用指纹**。
- **生成参数指纹**：Model、Sampler、Steps、CFG、尺寸及其他可比较参数。
- **Seed**：保留精确值和缺失状态，缺失不能误判为相同。

关系组由索引动态推导；人工收藏集和系列继续使用明确成员关系，不能与自动关系混为一谈。

## 6. 第一阶段：三页面外壳、图库与图片详情

### 6.1 阶段目标

先改变应用的主要导航和图片浏览路径，使用户不再同时面对左、中、右、下四个工作区域。本阶段完成后，日常导入、浏览和查看单张图片不依赖右侧检查器。

### 6.2 开发范围

#### 应用外壳

- 使用 Lobe `SideNav` 和 `Layout` 建立统一外壳。
- 建立 `gallery`、`vibes`、`tags`、`settings` 页面状态。
- Vibe 库和 Tag 库在本阶段可以先使用可导航的阶段占位页，但不能显示成不可点击的假入口。
- 保留现有明暗主题、字号、密度和跨平台窗口行为。

#### 图库

- 全部作品使用响应式图片网格，不再使用固定左侧窄列表作为主要浏览方式。
- 支持导入按钮、PNG/ZIP 拖拽、搜索、排序、收藏、回收站、多选和批量操作。
- 收藏集、系列和实验作为图库二级视图或筛选，不进入一级 SideNav。
- 图片卡片默认只展示缩略图、名称、收藏状态和必要异常标识。
- 图片加载使用缩略图、懒加载和占位 Skeleton；大量图片不得一次解码原图。

#### 图片详情

- 点击图片进入独立详情页，而不是只替换中央舞台内容。
- 顶部提供返回、面包屑、图片名称、文件夹、复制和创建方案操作。
- 使用 Lobe `Tabs` 提供概览、Prompt、Vibe、元数据、关系与方案。
- 概览展示大图和关键生成摘要。
- Prompt 复用现有 Prompt 总览、分类、多选、翻译显示和复制能力。
- Vibe 复用现有 Vibe 查看与可用性提示。
- 元数据展示只读生成事实和原始 metadata。
- 原图详情默认只读；“创建方案”进入分支编辑状态，不能直接改写原图。
- 关系与方案在第三阶段前先以标准列表提供现有分支、结果和实验入口，不使用胶片。

#### 移除永久检查器

- Prompt、Vibe 和参数能力迁移完成后，删除永久右侧检查器的用户入口。
- 删除依赖右侧宽度的响应式逻辑。
- 保留 Context Menu、快捷键和批量操作的功能对等入口。

### 6.3 推荐复用的 Lobe 组件

- `SideNav`、`Layout`、`Tabs`
- `Grid`、`Image`、`PreviewGroup`
- `SearchBar`、`Segmented`、`Select`
- `Button`、`ActionIcon`、`DropdownMenu`、`ContextMenu`
- `Skeleton`、`Empty`、`Alert`、`Toast`

业务 CSS 只负责网格、间距和页面响应，不重新实现组件的完整颜色与交互状态。

### 6.4 阶段验收标准

- 可以在图库、Vibe 库、Tag 库和设置之间稳定切换。
- 图库网格在常见窗口宽度下不遮挡按钮、不挤压文字。
- 导入、搜索、多选、收藏集、回收站和右键菜单功能无回退。
- 图片详情能够完整查看大图、Prompt、Vibe 和元数据。
- 原图生成事实不可编辑；创建方案仍能进入分支工作流。
- 永久右侧检查器不再显示。
- 普通图片详情不显示底部胶片；现有分支和实验仍有标准列表入口。
- 返回图库恢复筛选和滚动位置。
- Windows 隐藏应用菜单，macOS 保留原生菜单习惯。

## 7. 第二阶段：Vibe 库、Tag 库与反向关联

### 7.1 阶段目标

把 Vibe 和 Tag 从图片检查器中的附属内容提升为独立资源库，并建立“资源 → 图片”和“图片 → 资源”的双向查询。

### 7.2 Vibe 库

#### 列表页

- 默认按来源 PNG 分组，同一来源派生的多个 `.naiv4vibe` 变体放在组内。
- 支持搜索、手动分类、归档、缺少来源 PNG 筛选和 Information 状态筛选。
- 来源卡片显示参考图、编码数量、已知缓存位置和关联图片数。
- 无来源 PNG 的编码使用明确占位和警告，不伪造缩略图。

#### 详情页

- 展示来源 PNG、文件路径和打开文件夹操作。
- 展示 `.naiv4vibe` 编码变体及其模型、Information Extracted 和验证来源。
- 展示 Reference Strength 等项目内使用参数，但不把它们误认为编码文件属性。
- 使用图片网格反向展示引用该来源或使用指纹的图片。
- 支持从相关图片进入图片详情，并保留返回上下文。
- 继续执行“未知位置提示重新上传或提取、已缓存位置可复用”的规则。

### 7.3 Tag 库

#### 列表页

- 使用可虚拟化的紧凑列表或表格，不把全部 Tag 渲染成无边界标签云。
- 字段包含原始 Tag、翻译、分类、使用次数、关联图片数和异常状态。
- 支持搜索、分类、使用次数、翻译状态、AI/人工来源和隐藏状态筛选。
- 默认排除 `hidden` 和 `problematic`，并提供明确的“隐藏/异常”筛选入口。

#### 详情页

- 原始 Tag 使用只读组件展示，不出现重命名、删除或替换按钮。
- 允许编辑翻译、分类、备注、搜索别名和显示状态。
- 展示 Base/Character、Prompt/Undesired 等使用范围统计。
- 使用图片网格反向展示所有关联图片。
- 显示问题原因和原始语法样例，但不能自动修复图片 Prompt。

#### 缓存规则

- 相同 Tag 优先复用人工翻译和分类，其次复用 AI 缓存。
- 人工校正覆盖 AI 展示结果，但保留来源与更新时间。
- 图片重新解析或 Prompt 新增 Tag 时复用词典，不重复调用 AI。
- 词典数据更新不能触发图片 metadata 写入。

### 7.4 阶段验收标准

- 可以从图片详情进入 Vibe/Tag 详情，也可以从 Vibe/Tag 反向进入图片详情。
- 同来源 Vibe 能正确归组，不同 Information/Strength 不被错误合并。
- 缺少来源 PNG、未知 Information、用户设置和已验证状态区分清楚。
- Tag 原文在所有词典编辑流程中保持不变。
- 问题 Tag 默认隐藏，但可查找、可诊断，复制原始 Prompt 时不丢失。
- 翻译和分类缓存可以跨图片复用，人工校正优先。
- 上千图片和大量 Tag 下，搜索、筛选和滚动保持可用。

## 8. 第三阶段：关系分组、方案迁移与胶片退役

### 8.1 阶段目标

用可解释的图片关系组替代线性胶片。关系页围绕当前图片回答“还有哪些图与它共享什么信息、哪些字段发生了变化”，并把分支、实验和对比迁移到标准页面中。

### 8.2 自动关系组

至少支持以下关系：

1. **同 Prompt，不同 Seed**。
2. **同 Base Prompt，Character 不同**。
3. **同 Character，Base Prompt 不同**。
4. **同 Seed，Prompt 不同**。
5. **同完整 Prompt，其他生成参数不同**。
6. **同 Vibe 来源或同 Vibe 使用配置**。
7. **单变量对比候选**：除一个字段外，其余可比较字段一致。
8. **人工系列与实验**：明确标记为人工关系，不与自动推导混合。

### 8.3 关系命名与解释

组标题必须同时说明固定项与变化项，例如：

- `同 Prompt · Seed 不同`
- `同 Base Prompt · Character 不同`
- `同 Character · Base Prompt 不同`
- `同 Prompt · Seed / Vibe 有变化`
- `仅 Vibe 不同 · 单变量对比候选`

组内同时显示图片数量、固定字段摘要和变化字段徽标。混合变量关系只能称为“关联”或“混合变量”，不能暗示单一变量造成视觉差异。

### 8.4 分组排序与去重

- 少于两张图片不创建自动关系组。
- 单变量精确关系优先于多变量宽泛关系。
- “同完整 Prompt”优先于“同 Base Prompt”等包含范围更大的关系。
- 成员完全相同的多个关系合并展示，标题列出多个命中条件。
- 默认展开最相关的 2–3 组，其余使用 Lobe `Collapse` 折叠。
- 每组默认展示前 6–8 张图片，通过“查看全部”进入完整网格。
- 同一图片可以出现在不同语义组中，但同一组内不能重复。
- AI 相似度以后如加入，必须单独标记为“相似”，不能混入精确“相同”关系。

### 8.5 展示与对比

- 所有关系组使用标准响应式 `Grid`，不使用横向胶片。
- 图片卡片显示缩略图和相对当前图片的差异摘要。
- 组内支持多选 2–4 张图片进入独立对比页面。
- 对比页面继续支持视觉对比、参数差异、同步缩放和拖动。
- 对比页面不恢复底部胶片；参与对比的图片使用普通网格/列表管理。

### 8.6 分支、方案与实验迁移

- 图片详情的“关系与方案”展示原图、分支方案、待上传结果和已匹配结果。
- 无图片的待生成方案使用标准占位卡片，不使用编号胶片块。
- 上传结果、metadata 核对、不匹配子分支和多结果能力继续保留。
- 实验详情使用独立页面展示基准图、成员网格、固定字段和变化字段。
- 实验成员排序使用标准可排序网格，并继续提供键盘替代操作。

### 8.7 删除旧实现

功能迁移和数据核对完成后：

- 删除 `version-rail`、`version-strip`、`version-card` 等胶片组件和样式。
- 删除旧图片/Prompt 中央模式切换中已被详情 Tabs 替代的状态。
- 删除永久 Inspector 及其布局列宽和容器查询。
- 删除只服务旧布局的响应式覆盖、拖拽预览和兼容状态。
- 更新旧设计文档中的“实验胶片”表述，说明已被实验网格与关系页替代。

### 8.8 阶段验收标准

- 关系组能够准确区分相同与变化字段，缺失 metadata 不误判。
- 大组不会造成页面无限横向滚动或大量重复内容。
- 单变量候选、混合变量关联和人工系列在视觉与文案上可区分。
- 分支创建、待生成、结果上传、metadata 核对和不匹配处理无功能回退。
- 实验创建、成员管理、基准图和独立对比页面无功能回退。
- 底部胶片在所有页面消失，相关旧代码与 CSS 被删除。
- 普通工作区只保留 SideNav、页面标题、当前页面内容和必要操作。

## 9. 跨阶段工程要求

### 9.1 数据迁移

- 所有数据库变更使用可重复、可回滚前进的版本迁移。
- 先建立新索引并双读验证，再切换界面，最后删除旧派生状态。
- 不复制图片原始 Prompt 作为可编辑 Tag 词典数据。
- 迁移前后核对图片数、Tag 出现数、Vibe 编码数、分支数、系列数和实验数。

### 9.2 性能

- 图库、相关图片和 Tag 列表使用分页、虚拟化或渐进加载。
- 图片只加载缩略图；打开预览时才读取大图。
- 指纹在导入或 metadata 更新时增量计算，不能每次打开关系页扫描全库。
- 为 Tag 查找、图片关联、Vibe 来源和关系指纹建立数据库索引。
- 关系查询需要稳定排序，避免每次打开顺序变化。

### 9.3 可访问性与输入方式

- SideNav、Tabs、图片卡片、多选和分组展开支持完整键盘操作。
- 所有图标按钮提供可访问名称和 Tooltip。
- 拖拽始终有菜单或键盘替代方案。
- 关系差异不能只用颜色表达，必须显示字段名称或图标。
- 焦点从列表进入详情、返回列表和切换 Tabs 后落点可预测。

### 9.4 跨平台

- 共享 npm 脚本不加入平台专属语法。
- Ctrl/Cmd、Alt/Option、路径和文件夹操作使用平台适配层。
- Windows 验证窗口菜单、拖拽、文件夹打开和缩放；macOS 验证原生菜单、快捷键、窗口行为和文件授权。
- 平台敏感问题按 `coordination/README.md` 记录交接，不为普通界面工作创建无效交接。

### 9.5 每阶段质量门槛

每个阶段完成后都必须：

1. 运行 `npm ci`、`npm test` 和 `npm run build`。
2. 在深色和浅色主题下检查主要页面。
3. 检查窄窗口、默认窗口和大屏布局。
4. 验证导入、返回、搜索、筛选、多选和右键菜单。
5. 验证不会写回图片原始 metadata 或 `.naiv4vibe`。
6. 提交并推送当前开发分支。

## 10. 明确不在本轮三阶段中的内容

- 不使用 AI 模糊相似度替代精确关系指纹。
- 不修改图片中的原始 Tag 文本。
- 不在 Tag 库提供批量 Prompt 改写。
- 不重新编码或修改 `.naiv4vibe`。
- 不为桌面端使用移动端底部 `TabBar`。
- 不继续完善旧胶片交互或为其开发新的自定义组件。
- 不因新页面架构删除分支、实验、系列或对比的数据能力。

## 11. 最终完成定义

三个阶段全部完成后，应用应满足：

- 用户启动应用后只面对图库、Vibe 库和 Tag 库三个明确入口。
- 图库以图片网格为主，图片详情负责展示单张图片的全部事实和关系。
- Vibe 和 Tag 都可以反向查看关联图片。
- Tag 原文和图片生成事实不可变，用户注释与原始数据边界明确。
- 关系页使用可解释、可折叠的图片网格分组，不出现底部胶片。
- 永久右侧检查器、底部胶片及其旧布局代码完全删除。
- 专业能力通过详情页、关系页和对比页渐进呈现，不再同时挤在一个工作区。
- Windows 与 macOS 保持相同数据语义，并遵循各自平台交互习惯。
